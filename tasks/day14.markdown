## 任务 1：先把你当前项目的真实结构说清楚

先不要写理想结构，先看你当前代码里真实存在什么。

你现在 `notes/` 目录大致是：

```text
notes/
├─ admin.py
├─ apps.py
├─ middleware.py
├─ models.py
├─ permissions.py
├─ serializers.py
├─ services.py
├─ tests.py
├─ urls.py
├─ utils.py
└─ views.py
```

这张结构图不是“摆设”，而是帮助你先认清当前盘面。

下面把这些文件逐个拆开讲。

### 1.1 `models.py`

这个文件负责：

- 定义数据库结构
- 定义字段
- 定义模型之间的关系

例如你现在项目里的：

- `Note`
- `Tag`
- `Comment`

都属于这一层。

这层只回答一个问题：

- 数据长什么样

它不负责：

- 处理请求
- 拼响应
- 权限判断

---

### 1.2 `serializers.py`

这个文件负责：

- 输入校验
- 输出结构定义

你当前项目里这一层已经很明确了，比如：

- `NoteListSerializer`
- `NoteDetailSerializer`
- `NoteCreateSerializer`
- `NoteUpdateSerializer`
- `NoteListQuerySerializer`

也就是说：

- 列表怎么返回
- 详情怎么返回
- 创建时允许传什么
- 更新时允许传什么
- 查询参数允许怎么传

这些都放在这里。

这层不应该负责：

- ORM 查询
- 复杂业务编排
- HTTP 路由分发

---

### 1.3 `services.py`

这个文件负责：

- ORM 查询
- 数据创建、更新、删除
- 事务控制

你当前项目里这一层对应的是：

- `list_notes(...)`
- `get_note(...)`
- `get_note_detail(...)`
- `create_note(...)`
- `update_note(...)`
- `delete_note(...)`

这层回答的问题是：

- 数据怎么查
- 数据怎么改
- 哪些操作要放在事务里

它不应该负责：

- 拿 `request.data`
- 直接返回 `Response`

---

### 1.4 `permissions.py`

这个文件负责：

- 权限规则

例如你现在项目里的：

- `IsAuthorOrReadOnly`

这层是在回答：

- 谁能读
- 谁能写
- 谁能改自己的对象

---

### 1.5 `views.py`

这个文件负责：

- 接收请求
- 调 serializer
- 调 service
- 做请求编排
- 返回响应

你现在项目里的主线 view 是：

- `NoteListAPIView`
- `NoteDetailAPIView`
- `NoteCreateAPIView`

这层应该做的是“调度”，而不是把 ORM 细节一股脑堆进去。

---

### 1.6 `urls.py`

这个文件负责：

- URL 和 view 的绑定关系

例如：

- 哪个 URL 进入列表接口
- 哪个 URL 进入详情接口
- 哪个 URL 进入创建接口

---

### 1.7 `tests.py`

这个文件负责：

- 用测试去验证接口行为

它回答的问题不是：

- “代码长什么样”

而是：

- “这套接口是不是真的按预期工作”

---

### 1.8 `utils.py`

这个文件在你当前项目里，实际上主要放的是：

- `NotePagination`

也就是说，它目前承担的是：

- 分页配置

这里就出现了一个很典型的 Day 14 收尾问题：

- 功能没错
- 但文件命名是不是足够清楚

如果一个文件里以后只放分页类，那么比起 `utils.py`，很多项目会更倾向写成：

- `pagination.py`

原因不是“必须”，而是：

- `utils.py` 太泛
- `pagination.py` 一眼就知道职责

---

### 1.9 这张目录结构图完成了什么功能

它完成的不是“画图好看”，而是：

1. 让你先认清当前项目真实结构
2. 让你知道每层职责边界
3. 让后面的整理动作有参照物

## 任务 2：给当前项目写一句总原则

今天建议你明确写下这几条总原则：

```text
models 不处理 HTTP
services 不处理 request/response
serializers 负责输入输出，不做复杂业务编排
views 负责请求编排，不堆 ORM 细节
permissions 负责权限规则
tests 负责验证行为
```

下面逐行拆开讲。

### 2.1 `models 不处理 HTTP`

这句话的意思是：

- 模型层负责数据结构
- 不应该直接关心 URL、请求方法、状态码

---

### 2.2 `services 不处理 request/response`

这句话的意思是：

- service 层负责业务动作和 ORM
- 不应该依赖 `request`
- 不应该直接 `return Response(...)`

如果 service 里开始出现很多 HTTP 逻辑，边界就乱了。

---

### 2.3 `serializers 负责输入输出，不做复杂业务编排`

这句话的意思是：

- serializer 很适合做字段校验和表示层定义
- 但不应该承担整套业务流程编排

否则你以后读代码时会发现：

- 校验逻辑、创建逻辑、事务逻辑、权限逻辑都挤在 serializer 里

---

### 2.4 `views 负责请求编排，不堆 ORM 细节`

这句话的意思是：

- view 要负责请求入口和流程调度
- 但不应该塞满 `Note.objects.filter(...).select_related(...).prefetch_related(...)`

否则你之前为什么拆 service，就又白拆了。

---

### 2.5 `permissions 负责权限规则`

这句话的意思是：

- “谁能做什么”这种规则尽量集中

不要把：

- 一半权限判断写在 view
- 一半权限判断写在 service
- 另一半写成临时 `if`

---

### 2.6 `tests 负责验证行为`

这句话的意思是：

- 测试不是摆设
- 它是在帮你守住这条主线不被后面改乱

---

### 2.7 这组总原则完成了什么功能

它完成的是：

1. 把项目分层写成清楚的约定
2. 让你以后写新功能时知道该往哪一层放
3. 减少“今天写在这里，明天又搬到那里”的摇摆

---

## 任务 5：建立 Schema / OpenAPI 的最小认知

今天不要求你把整套文档系统接完，但你必须至少知道：

- 什么是 Schema
- 什么是 OpenAPI
- DRF 可以自动生成 API 描述

先看一段最小示例：

```python
from django.contrib import admin
from django.urls import include, path
from rest_framework.schemas import get_schema_view


urlpatterns = [
    path("admin/", admin.site.urls),
    path("", include("notes.urls")),
    path(
        "schema/",
        get_schema_view(
            title="StudyNotes API",
            description="API schema for the StudyNotes project",
            version="1.0.0",
        ),
        name="api-schema",
    ),
]
```

这段代码你今天不用马上接进项目，重点是看懂它在做什么。

下面逐行拆开讲。

### 5.1 `from rest_framework.schemas import get_schema_view`

这一行导入的是：

- DRF 提供的 Schema 视图工厂函数

你可以先把它理解成：

- “帮你生成一个专门输出 API 描述文档的 view”

这里的关键词是：

- `schema`
  表示接口结构描述

---

### 5.2 `path("schema/", ..., name="api-schema")`

这一行的意思是：

- 给项目新增一个 `/schema/` 路由

这里的：

- `"schema/"`
  表示访问路径
- `name="api-schema"`
  表示这条路由的名字

这行完成的功能是：

- 让你的项目有一个专门用来输出 API 描述的端点

---

### 5.3 `get_schema_view(...)`

这行的意思是：

- 创建一个 schema view

也就是说：

- 访问 `/schema/` 时
- 返回的不是普通业务数据
- 而是整个 API 的结构描述

---

### 5.4 `title="StudyNotes API"`

这个参数的意思是：

- 给这份 API 描述起一个标题

它是给谁看的：

- 给开发者看
- 给文档工具看

---

### 5.5 `description="API schema for the StudyNotes project"`

这个参数的意思是：

- 给这份 API 描述写一句说明

它通常用来告诉别人：

- 这是哪个项目的 API
- 这份文档大致描述什么

---

### 5.6 `version="1.0.0"`

这个参数的意思是：

- 给当前 API 说明一个版本号

它的作用不是“装饰”，而是：

- 方便协作时区分 API 版本
- 方便以后演进时做记录

---

### 5.7 这段 schema 路由代码完成了什么功能

它完成的是：

1. 在项目里新增一个 `/schema/` 路由
2. 让这个路由返回 API 描述信息
3. 给 API 描述附加标题、说明和版本号

这就是你今天要建立的最小认知：

- 现代 API 项目不只是“接口能跑”
- 它还应该能被描述、被阅读、被工具消费

---

## 任务 6：把 `Schema`、`OpenAPI`、`AutoSchema` 三个词区分开

今天这 3 个词你只要分清层级就够了。

### 6.1 `Schema`

你可以先把它理解成：

- “接口结构说明”

它回答的问题是：

- 有哪些接口
- 接口收什么参数
- 接口返回什么结构

---

### 6.2 `OpenAPI`

你可以先把它理解成：

- “描述 API 的一种通用标准格式”

也就是说：

- Schema 是“描述”
- OpenAPI 是“这种描述常用的一种标准”

所以别人说：

- “我们有 OpenAPI 文档”

通常就是在说：

- “我们的 API 描述遵循 OpenAPI 这种通用格式”

---

### 6.3 `AutoSchema`

你可以先把它理解成：

- DRF 里用于自动推断接口描述的一类机制

意思是：

- 你不需要完全手写每个接口的文档
- DRF 可以根据 view、serializer、字段等信息自动推断一部分结构

今天你不用把它写到项目里，但你要知道：

- DRF 的 schema 能力不是纯手写出来的
- 很多地方是“自动生成 + 必要时补充说明”

---

### 6.4 这 3 个概念区分完成了什么功能

它完成的是：

1. 让你不再把这些词混成一团
2. 让你知道 API 文档不是纯手写说明书
3. 让你对 DRF 的文档能力有最小认知

---

## 任务 8：规划下一阶段学习顺序

今天最后一个动作不是继续发散学新类，而是把下一阶段排个顺序。

如果你接下来仍然以“做 API 项目”为目标，比较合理的下一阶段可以是：

1. JWT 认证
2. 文件上传
3. 缓存和性能优化
4. Celery 异步任务
5. Docker 与部署
6. 日志与监控

下面把这个顺序简单解释一下。

### 8.1 `JWT 认证`

它解决的是：

- 前后端分离项目里更常见的认证方式

你现在已经理解了 Session 和权限，这一步很自然。

---

### 8.2 `文件上传`

它解决的是：

- 真实项目里图片、附件、头像等文件处理需求

---

### 8.3 `缓存和性能优化`

它解决的是：

- 接口变多后，怎么减少数据库压力
- 热点数据怎么更快返回

---

### 8.4 `Celery 异步任务`

它解决的是：

- 邮件发送
- 批量任务
- 耗时操作不阻塞请求

---

### 8.5 `Docker 与部署`

它解决的是：

- 项目怎么从“本地能跑”走向“可部署”

---

### 8.6 `日志与监控`

它解决的是：

- 出问题以后怎么定位
- 接口运行状态怎么观察

---

### 8.7 这份下一阶段计划完成了什么功能

它完成的是：

1. 给当前阶段一个清楚结束点
2. 给下一阶段一个稳定入口
