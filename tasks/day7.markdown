## 今天你必须先理解的 9 个概念

## 1. “本地能跑”和“可以部署”不是一回事

这是 Day 7 最重要的一句话。

很多项目在本地跑得很好，但一旦要部署，就会暴露很多问题，比如：

- `DEBUG=True`
- 敏感信息写死
- 主机名没配置
- 静态文件、日志、数据库连接没考虑清楚

所以今天的核心，不是让你一定要把项目真上线，而是让你开始知道：

- 真正的交付要考虑哪些东西

## 2. `DEBUG=False` 为什么重要

开发环境里开 `DEBUG=True` 很方便，因为：

- 报错页面详细
- 调试体验好

但生产环境必须认真对待这个配置。

因为如果继续开着：

- 可能泄露敏感信息
- 错误页面暴露内部结构

所以你今天必须建立一个基本意识：

- 开发环境和生产环境的配置不能混在一起不管

## 3. `ALLOWED_HOSTS` 是什么

你可以先把它理解成：

> 允许哪些主机名访问这个 Django 服务。

开发时常常是空列表或本地地址。  
但部署时要明确配置。

否则你项目部署出去以后，很容易一上来就卡在：

- 访问域名不被允许

## 4. 为什么敏感配置不应该写死在代码里

比如这些内容：

- 数据库密码
- `SECRET_KEY`
- 生产数据库地址

如果你把它们一直写在 `settings.py` 里，会有几个明显问题：

- 不安全
- 不方便切环境
- 不利于团队协作

所以你今天要开始建立环境变量意识。

## 5. `python manage.py check --deploy` 是什么

你可以先把它理解成：

> Django 提供的一个“部署前健康检查”。

它不会替你部署项目，但会提醒你一些典型问题，比如：

- 某些安全配置不合适
- 某些生产项还没配好

这条命令很值得尽早记住。

## 6. 性能优化不是“到很慢了再说”

真正好的习惯是：

- 在写接口的时候就开始形成性能意识

你前几天其实已经接触过了：

- `select_related`
- `prefetch_related`

今天要做的是：

- 真正选一个接口
- 解释为什么它可能有 N+1 风险
- 然后做一次有意识的优化

## 7. 什么是 N+1 查询问题

你可以先非常朴素地理解：

- 先查一批对象，1 次查询
- 再对每个对象分别查关联对象，N 次查询

合起来就变成：

- 1 + N 次查询

这就是典型的 N+1 问题。

比如你查 20 篇笔记，如果每篇笔记又单独查作者和标签，数据库请求次数就会快速变多。

所以：

- 外键常常考虑 `select_related`
- 多对多和反向关系常常考虑 `prefetch_related`

## 8. README 不是装饰，而是交付的一部分

很多人学项目时会忽略 README。  
但真实协作里，这个文件很重要。

至少应该说明：

- 这个项目是做什么的
- 怎么启动
- 需要哪些环境
- 主要接口有哪些

哪怕你现在是个人学习项目，也值得练一下这种习惯。

## 9. Day 7 的意义是“收尾能力”

前 6 天更多是在“构建能力”。  
今天更偏向“整理能力”和“交付能力”。

你会发现：

- 能写出功能是第一步
- 能把项目整理清楚，才能更容易继续维护

```python

```

数据库也可以类似这样：

```python
DATABASES = {
    "default": {
        "ENGINE": os.getenv("DB_ENGINE", "django.db.backends.postgresql"),
        "NAME": os.getenv("DB_NAME", "studynotes"),
        "USER": os.getenv("DB_USER", "postgres"),
        "PASSWORD": os.getenv("DB_PASSWORD", ""),
        "HOST": os.getenv("DB_HOST", "localhost"),
        "PORT": os.getenv("DB_PORT", "5432"),
    }
}
```

## 任务 6：看懂 `select_related` 和 `prefetch_related` 的分工

今天一定要把这两个再讲透一点。

## 6.1 `select_related`

适合：

- 外键
- 一对一

它通常是通过 SQL JOIN 一次性把关联对象一起查出来。

比如：

```python
Note.objects.select_related("author")
```

## 6.2 `prefetch_related`

更适合：

- 多对多
- 反向外键

它通常不是单纯靠一条 JOIN，而是：

- 额外查关联数据
- 然后在 Python 层拼起来

比如：

```python
Note.objects.prefetch_related("tags", "comments")
```

## 6.3 你今天至少要记住一句话

- 外键先想 `select_related`
- 多对多和反向关系先想 `prefetch_related`

## 今日复盘问题

今天结束后，试着不用看文档，自己回答这些问题：

1. 为什么 Django 项目要区分开发配置和生产配置？
2. `DEBUG=False` 和 `ALLOWED_HOSTS` 分别在保护什么？
3. 为什么数据库密码不应该长期写死在 `settings.py`？
4. 什么是 N+1 查询问题？
5. `select_related` 和 `prefetch_related` 分别适合什么关系？
6. 为什么 Day 7 要写 README 和后续待办，而不只是继续加功能？

如果你能比较顺畅地回答这些问题，说明你已经不只是在“学会 Django”，而是在开始学习“如何交付一个 Django 项目”。

## 可选加分任务

如果你今天状态不错，可以再做两个增强：

### 加分任务 1：尝试把配置读取改成环境变量形式

哪怕你先只改：

- `SECRET_KEY`
- `DEBUG`
- 数据库密码

也已经很有意义。

### 加分任务 2：为列表接口补一段性能优化说明

比如在 README 里写清楚：

- 为什么这里用了 `select_related("author")`
- 为什么这里用了 `prefetch_related("tags")`

这会让你的项目说明更像一个认真整理过的工程。
